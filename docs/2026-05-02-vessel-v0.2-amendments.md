# Vessel — v0.2 Amendments

**Date:** 2026-05-02
**Status:** Proposed amendments to [`spec.md`](spec.md) v0.1
**Origin:** Conversation-driven design pass. Changes here are candidates for absorption into a v0.2 canonical spec; the v0.1 body remains intact until that pass.

This document captures architectural decisions reached after v0.1 was published. Each section identifies the v0.1 text it modifies and the rationale for the change.

## 1. Distribution: container + thin shell

**Modifies:** §1 (What Vessel is), §3 (Architectural decisions, lock #4 "Tauri v2 shell"), §4 (Architecture diagram).

**Decision:** Vessel ships as a **container** (precise meaning TBD — see §1.1 below) hosting the services + SDKs + bundled UI manifests, plus a thin native shell window for the transparent always-on-top render surface. The shell is the visual boundary; the container is the engine.

**Rationale:** v0.1 modeled vessel as a single Tauri app that bundled everything. That was correct as a first sketch, but two pressures push toward a split:

- The UI shouldn't be a single fixed avatar overlay — it should be programmable so aspects (or other agents) can author their own interfaces against vessel's services. That requires a runtime that can host arbitrary UI manifests, not a hard-coded React tree.
- Common harnesses (Claude Code, Cursor, future tools) deserve dedicated UI defaults tuned to their surface. Bundling those alongside the avatar default fits a "platform of UIs" model better than a "single-app" model.

The container holds: STT service, TTS service, ChatSource SDK runtime, Interface manifest renderer, bundled default UIs (avatar, chat-only, harness-tuned variants). The thin shell hosts a native transparent always-on-top window pointing at the container's localhost UI endpoint.

### 1.1 "Container" — precise meaning open

The word "container" is load-bearing and ambiguous. Two candidate meanings:

- **OCI/Docker image.** Standard cloud-native packaging. Cross-platform. Cost: requires Docker Desktop on Win/Mac (heavy install), and audio device passthrough across the container boundary on Win/Mac is genuinely fiddly (Linux PulseAudio/PipeWire works; Mac/Win do not pass mic/speaker natively without WSL gymnastics).
- **Self-contained bundle.** Single installer (Tauri app + sidecar binaries + bundled web assets). Not a "container" in the OCI sense — but distributable as one artifact, with the same conceptual split between "the engine" and "the shell."

This needs to resolve before Phase 1 redefinition. Audio I/O is the deciding constraint: if Docker can't pass mic/speaker cleanly on the platforms that matter, Docker isn't viable as the distribution unit and the self-contained-bundle reading is the right answer. Phase 0 spike: confirm audio passthrough on Win/Mac/Linux Docker before committing.

## 2. ChatSource grows to carry UI manifests (no separate Interface SDK)

**Modifies:** §5 (Vessel SDK), §5.1 (ChatSource interface), §9 (Attention mechanism).

**Decision:** ChatSource is the **single bidirectional protocol** between vessel and a backend. UI manifests are a payload type carried over the same connection, not a separate SDK.

**Rationale:** An earlier framing of "two SDKs — ChatSource for chat, Interface SDK for UI" introduced an unnecessary boundary. Both surfaces are emitted by the same backend, consumed by the same vessel runtime, and need to be sequenced together (e.g. "render this UI then say this thing" requires shared ordering). Splitting them creates a coordination problem and doubles the adapter authorship cost without benefit.

Concretely, the v0.1 `BackendEvent` discriminated union extends:

```typescript
export type BackendEvent =
  | { type: "message"; payload: { recipient: string; from: string; text: string } }
  | { type: "attention"; payload: { from: string; urgency: "normal" | "critical" } }
  | { type: "thread_changed"; payload: { thread_id: string } }
  // v0.2 additions:
  | { type: "ui.manifest"; payload: UIManifest }
  | { type: "ui.event"; payload: { manifest_id: string; event: UIEvent } }
  | { type: "turn.begin"; payload: { turn_id: string; recipient: string } }
  | { type: "turn.end"; payload: { turn_id: string; recipient: string } };
```

`UIManifest` is a structured description of what to render (constrained component vocabulary, declared capability requirements). `UIEvent` is a user interaction with a rendered manifest, sent back to the backend. Adapters that don't speak UI manifests simply never emit them; vessel falls back to the avatar default.

The v0.1 split between `subscribe()` and `send()` remains: `send()` delivers user input to the backend (chat or UI event); `subscribe()` is the push channel for everything backend-initiated.

### 2.1 Trust model for AI-authored UIs

If a backend (e.g. an aspect) emits a `ui.manifest`, vessel is rendering content authored by an AI. Three properties the runtime must enforce:

- **Constrained component vocabulary.** Manifest references named components from a vessel-shipped registry, not raw HTML/JS. Forbids arbitrary script execution.
- **Declared capability scope.** Manifest declares what it needs (mic, file picker, clipboard, display size) and the runtime gates access. No silent capability acquisition.
- **User inspection / approval before first render.** Per-aspect, sticky after first approval; revocable.

Without these, a hostile aspect can render a phishing prompt that looks native. Open question: vocabulary spec lives where (in vessel-sdk, or as a separate `vessel-ui-vocab` package)?

## 3. MCP is not the right protocol for ChatSource

**Modifies:** §5 (Vessel SDK), no v0.1 text directly — clarifies what was implicit.

**Decision:** ChatSource is a **bidirectional, push-capable** protocol — WebSocket or SSE-with-uplink. **Not MCP.**

**Rationale:** MCP is structured as agent-initiated tool calls — the agent (host) drives, the server responds. Server-initiated traffic exists (`notifications/*`, `sampling/createMessage`) but is auxiliary. Crucially, **MCP cannot wake an agent and hand it a new turn** — it assumes the agent already has a turn underway.

ChatSource needs push capability on both sides:
- User → backend: "here's a new message, take a turn" (the wake direction).
- Backend → user: response stream, attention calls, presence updates, UI manifests, all unsolicited.

MCP can't carry the wake direction cleanly. Workarounds (polling, inverting topology) defeat its purpose. The `send_comms`-as-tool pattern that motivated this exploration was a real solution to the v0.1-era constraint, but it leaks at the seams: agents trained on assistant-message defaults forget to call the tool, and the system needs PTY-scraping fallbacks to catch the leakage.

The right separation:

- **ChatSource = the conversation channel.** Push-capable both directions. Wake works because messages flow either way at any time. WS or SSE-with-uplink as transport. **Not MCP.**
- **MCP = optional capability surface.** Vessel exposes mic/speaker/avatar control as MCP tools the harness *can* reach for during its turn. Pull-shaped, fits MCP's grain. Optional — adapters that don't care about fine-grained control just use ChatSource.

This mirrors nexus's own decision to build a WS-first transport (`/connect`, `turn` frames, `chat.deliver`) rather than build on MCP. Same problem, same answer.

## 4. Build order: chains-first, UI integration second

**Modifies:** §11 (Build plan), specifically Phase 1 ordering.

**Decision:** Build the input chain and output chain as headless CLIs first; integrate into the Tauri/VRM shell as a later step. Run a parallel Tauri+VRM render-proof spike in case the transparent overlay has cross-platform surprises, but do not gate chain work on it.

**Rationale:** Chains are where the *unknowns* live (STT quality, TTS+lipsync timing, ChatSource adapter shape, end-to-end latency); the UI is where the *certainties* live (Tauri, Three.js, VRM are well-understood). Phase 1 in v0.1 builds the certainties first, which is backwards from a risk-management perspective.

### 4.1 Revised phase order

1. **Input chain (CLI).** Push-to-talk binding + STT + cleanup-LLM pass. Bench against existing tools (Glaido, Superwhisper) on the same audio samples. Produces `text` from `audio` on stdout.
2. **Output chain (CLI).** TTS + phoneme timing + audio playback. Reads text on stdin; produces audio + phoneme JSON. Bench end-to-end latency.
3. **Glue: ChatSource SDK + Nexus reference adapter (CLI).** Full text-only conversation: speak → backend → speech. Still no UI. Validates SDK shape under real backend pressure.
4. **Parallel Tauri/VRM spike** (runs alongside 1–3). Confirm transparent overlay + VRM render works on macOS and Linux (open question §14.1). Throwaway-quality is fine.
5. **Integrate.** Chains move into the container, VRM gets driven by phoneme timing from step 2, default UI manifest renders the avatar in the Tauri shell, ChatSource adapter from step 3 is the connection.

Steps 1–3 each produce a usable artifact. Step 4 de-risks the only piece that can't be done headless. Step 5 is the merge.

## 5. STT cross-platform strategy

**Modifies:** §3 (lock #5 "whisper.cpp via Rust backend"), §12 (STT pipeline), §14 (Open questions).

**Decision:** `whisper.cpp` (via `whisper-rs`) is the cross-platform default. The STT engine is **pluggable behind an `STTEngine` trait** so per-platform accelerators can drop in later (WhisperKit on macOS, ONNX+DirectML on Windows). The cleanup-LLM pass after STT is part of the input chain, **not** part of the STT engine.

**Rationale:** Cross-platform is a hard product constraint. Apple-Silicon-optimised paths (WhisperKit, macparakeet, Apple SpeechAnalyzer) deliver best-in-class on Mac but rule out Win/Linux. `whisper.cpp` is the only mature engine that runs well on all three. Acceptance: vessel won't beat Mac-native competitors on Mac at v1; it will be acceptable on Mac and *the only option that works* on Win/Linux.

The cleanup-LLM pass is the bigger lever than STT engine choice. Raw Whisper transcripts → "ready-to-send" text is mostly a prompting problem (Claude Haiku, or a small local model). Trivially cross-platform. This is also the differentiator vs. tools like Glaido — Glaido's moat is the cleanup pass, not the underlying STT.

### 5.1 Tension with §15 non-goal "Modifying recipient behavior or prompt context"

The v0.1 non-goal says vessel never modifies what the recipient sees. Strict reading: **this includes STT cleanup**, since the user's spoken input arrives at the recipient differently from how it was uttered.

**Resolution proposed:** distinguish:
- **Transcription cleanup** — removing filler words ("um", "uh"), inserting punctuation, fixing whisper homophone errors. *Does* count as "what the user said" in any reasonable sense; the user's intent is preserved. **Allowed.**
- **Message rewriting / context injection** — vessel adding system prompts, persona instructions, or surrounding context. **Forbidden** (preserves v0.1 non-goal).

The line is intent-preservation. Spec wording in §15 should make this distinction explicit; the current text reads as if all modification is forbidden, which would forbid even basic punctuation insertion.

## 6. Glaido as comparator (informational, no spec change)

**Adds context to:** §11 Phase 1/5 acceptance criteria; §14 (Open questions).

Glaido (https://glaido.com) is a native-Mac voice-typing tool that demonstrates the feel-quality bar for "single-shortcut, sub-second, ready-to-send transcription" on a 2026 laptop. It is not a competitor (Mac-only; vessel's primary platforms are Win/Linux), and not a building block (no SDK; closed cloud backend). It is a proof-of-existence for what the input chain should *feel* like.

Phase 5 acceptance should include a calibration test: same audio sample through vessel's input chain and Glaido. Vessel's output should be in the same league. If noticeably worse, gap → tune cleanup prompt, swap STT model, or accept and move on.

## 7. Open questions added in v0.2

These join the v0.1 §14 list:

6. **Container packaging — OCI vs self-contained bundle.** Audio device passthrough on Win/Mac Docker is the deciding constraint. Phase 0 spike before Phase 1 redefinition.
7. **UI manifest vocabulary spec — where it lives.** In `@nexus-cw/vessel-sdk` (couples manifest spec to SDK lifecycle) or `@nexus-cw/vessel-ui-vocab` (independent versioning, safer for backend/runtime version drift)?
8. **Manifest capability gating — UX for first-render approval.** Per-aspect sticky approvals like passkeys, or per-session like browser permissions, or hybrid?
9. **Cleanup-LLM provider when offline.** Edge TTS already requires net; cleanup-LLM does too if it's Claude Haiku. Local fallback model (Llama 3.x small / Phi-mini) feasibility for offline use case?
10. **Glaido cleanup-pass benchmark target.** What latency / quality bar does vessel commit to before Phase 5 ships?

## 8. What this document does *not* change

For the avoidance of doubt, the following v0.1 decisions are unaffected:

- Tauri v2 as the shell technology (it is now the *shell over* the container, not the whole app).
- VRM model format and Three.js + `@pixiv/three-vrm` rendering.
- Edge TTS primary + Piper fallback for TTS.
- rhubarb-lip-sync for phoneme timing.
- `whisper-rs` as the STT default engine.
- Single-user at v1.
- Speech queue model, attention mechanism, active-thread filtering.
- All non-goals in §15 except the §5.1 transcription-cleanup clarification above.
